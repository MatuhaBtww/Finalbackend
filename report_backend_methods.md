# РЕФЕРАТ

В работе рассмотрены проектирование и реализация backend-части информационной системы «Парикмахерская». Серверная часть разработана на языке Python с использованием фреймворка Django и библиотеки Django REST Framework. Система предназначена для автоматизации записи клиентов на услуги, управления ролями пользователей, расписанием мастеров, статусами записей, оплатой, журналом действий и аналитикой риска неявки клиента.

Цель работы состоит в создании работоспособного backend-приложения с базой данных, REST API, разграничением прав доступа и AI-эндпоинтом. В ходе работы были выполнены анализ предметной области, проектирование модели данных, реализация сущностей и связей базы данных, настройка аутентификации, разработка API для основных операций и внедрение модуля прогнозирования вероятности неявки клиента.

Результатом работы стал backend-проект, включающий сущности пользователей, ролей, прав доступа, услуг, расписаний мастеров, записей, транзакций, AI-данных и журнала действий. Реализованы аутентификация по JWT, CRUD-операции, фильтрация, бизнес-операции по записи и отдельный AI-эндпоинт для анализа риска неявки. Практическая значимость работы заключается в автоматизации процессов салона и снижении числа ошибок при записи клиентов.

Особый интерес представляет использование аналитического модуля, который позволяет заранее оценить риск неявки клиента и тем самым помогает администратору принимать более обоснованные решения.

Цель работы — спроектировать и реализовать backend информационной системы «Парикмахерская» на Python и Django.

Объектом исследования является процесс записи клиентов в парикмахерскую. Предметом исследования выступают методы проектирования и разработки backend-информационной системы для автоматизации этого процесса.

# 1. ТЕХНИЧЕСКОЕ ЗАДАНИЕ

## 1.1. Цель разработки

Цель разработки состоит в создании backend-части веб-приложения «Парикмахерская», которая обеспечивает хранение и обработку данных предметной области, предоставляет программный интерфейс для взаимодействия с системой и реализует базовую бизнес-логику салона. Серверная часть должна работать как отдельное приложение, взаимодействующее с базой данных и клиентской стороной по HTTP с передачей данных в формате JSON.

## 1.2. Назначение системы

Система предназначена для автоматизации следующих задач:

1. регистрация и авторизация пользователей;
2. хранение информации о ролях и правах доступа;
3. управление услугами парикмахерской;
4. ведение расписания мастеров;
5. создание и сопровождение записей клиентов;
6. регистрация оплаты;
7. ведение журнала действий пользователей;
8. выполнение AI-анализа вероятности неявки клиента.

Таким образом, система решает задачу структурированного учета и снижает зависимость от ручного ведения записей.

## 1.3. Профиль предметной области

В реальной работе парикмахерской значительная часть операций часто выполняется вручную: администратор принимает звонок или сообщение, уточняет удобное время, проверяет занятость мастера, подбирает услугу, фиксирует запись и при необходимости меняет статус визита. Такой подход создает ряд проблем. Во-первых, возрастает риск записи двух клиентов на одно время. Во-вторых, информация может храниться в разных источниках: в тетради, мессенджере, таблице или в памяти администратора. В-третьих, затрудняется контроль оплат и истории действий. Наконец, невозможно получать аналитику по неявкам клиентов и эффективности загрузки мастеров.

После внедрения системы основные шаги автоматизируются. Пользовательские данные хранятся централизованно, мастер имеет расписание, запись создается только при отсутствии конфликта по времени, статусы обслуживания фиксируются формально, а действия пользователей сохраняются в журнале. Дополнительно становится возможным прогнозировать риск неявки клиента по ряду признаков.

## 1.4. Пользователи, роли и права доступа

В системе выделены следующие роли:

1. администратор;
2. мастер;
3. клиент.

Администратор управляет пользователями, ролями, справочниками, статусами, услугами, расписанием и журналом действий. Мастер работает со своим расписанием и связанными с ним записями. Клиент может регистрироваться, авторизовываться, просматривать доступные услуги и создавать собственные записи.

Для реализации разграничения доступа в проекте используются сущности `Role` и `AccessRight`, а также набор permission-классов на уровне API.

## 1.5. База данных и сущности

В базе данных реализованы следующие основные сущности:

1. `User` — пользователь системы;
2. `Role` — роль пользователя;
3. `AccessRight` — право доступа;
4. `Service` — услуга;
5. `Status` — статус записи;
6. `MasterSchedule` — расписание мастера;
7. `Appointment` — запись клиента;
8. `Transaction` — транзакция оплаты;
9. `Aidata` — данные AI-анализа;
10. `AuditLog` — журнал действий.

Для сущностей определены первичные и внешние ключи, ограничения уникальности, обязательность заполнения и связи между таблицами. Центральной сущностью является запись `Appointment`, которая связывает клиента, мастера, услугу и статус.

## 1.6. Бизнес-процессы

Система поддерживает как минимум три сквозных бизнес-процесса.

Первый процесс — создание записи клиента. На входе используются данные клиента, услуга, мастер и желаемое время. На этапе обработки система проверяет валидность данных, доступность мастера и отсутствие пересечений. Результатом является новая запись в базе данных со статусом и временем окончания, рассчитанным по длительности услуги.

Второй процесс — сопровождение записи администратором. На входе используется уже существующая запись. В ходе обработки администратор подтверждает визит, отменяет его или завершает обслуживание. Результатом становится изменение статуса записи и соответствующая запись в журнале действий.

Третий процесс — оплата услуги. На входе используется идентификатор записи и сведения об оплате. В ходе обработки создается объект транзакции и обновляется платежный статус записи. Результат фиксируется в БД и в журнале аудита.

## 1.7. Задача искусственного интеллекта

В качестве AI-функции в системе реализовано прогнозирование вероятности неявки клиента на запись. Эта функция встроена в бизнес-процесс сопровождения записи и может использоваться перед подтверждением визита или при анализе клиентской дисциплины.

Входными признаками являются:

1. количество предыдущих посещений;
2. число поздних отмен;
3. число неявок;
4. число дней до записи;
5. факт предоплаты;
6. показатель лояльности клиента.

Выходом является вероятность неявки, представленная в процентах. Результат сохраняется в базе данных и возвращается через API в понятном пользователю виде.

## 1.8. Методология разработки

Разработка велась по каскадной модели жизненного цикла. Сначала был выполнен анализ предметной области и сформированы требования, затем спроектирована структура базы данных и серверной архитектуры, после чего была реализована серверная логика и проведено тестирование. В заключительной части были подготовлены документация и итоговый отчет.

## 1.9. Архитектурные и технологические требования

Система построена по клиент-серверной архитектуре. Пользователь взаимодействует с системой через интерфейс или API-клиент, основная логика выполняется на сервере, а данные хранятся в базе данных. В качестве технологий выбраны Python, Django, Django REST Framework и JWT-аутентификация. В проекте прослеживаются элементы паттернов MVC, Repository и Service Layer: модели отвечают за структуру данных, viewset-ы и сериализаторы — за взаимодействие с API, а бизнес-операции сосредоточены в серверной логике.

## 1.10. Требования к качеству и безопасности

Система должна обеспечивать:

1. корректную валидацию входных данных;
2. разграничение прав доступа по ролям;
3. журналирование действий пользователей;
4. устойчивую обработку ошибок с понятными сообщениями;
5. безопасное хранение паролей в виде хэша;
6. централизованное хранение данных в структурированном виде.

## 1.11. Разработка архитектуры базы данных и реализация системы

Архитектура разрабатываемой системы строится вокруг разделения ответственности между данными, серверной логикой и средствами взаимодействия с пользователем. Даже если в рамках данной работы подробно реализуется именно backend, система изначально проектируется как решение с разделенными компонентами: пользовательский интерфейс рассматривается как отдельный клиент, серверная часть реализуется как отдельное приложение с API, а база данных выступает самостоятельным хранилищем.

С точки зрения компонентов архитектура выглядит следующим образом:

1. клиентский уровень — браузерный интерфейс, Swagger UI или другой клиент, который отправляет HTTP-запросы;
2. серверный уровень — Django-приложение, реализующее бизнес-логику, проверку данных, авторизацию, права доступа и AI-обработку;
3. уровень хранения данных — реляционная база данных SQLite, содержащая сущности предметной области, журнал действий и AI-результаты.

Взаимодействие компонентов организовано по клиент-серверной схеме. Пользовательский интерфейс или тестовый клиент обращается к серверу по HTTP. Сервер принимает запрос, выполняет валидацию, обращается к моделям и базе данных, при необходимости записывает действия в аудит и возвращает результат в формате JSON. Такой подход позволяет отделить интерфейс от логики обработки данных и при необходимости заменить клиентскую часть без переработки серверного ядра.

В проекте применяются паттерны проектирования, требуемые методичкой. Паттерн MVC проявляется в разделении моделей данных, представлений API и маршрутизации. Паттерн Repository реализуется через использование ORM и queryset-ов как слоя доступа к данным. Паттерн Service Layer выражается в том, что бизнес-операции не сводятся к простому CRUD, а оформлены как отдельные действия серверной логики: подтверждение записи, отмена записи, завершение записи, проведение оплаты и прогнозирование неявки.

Роли, журнал действий, бизнес-процессы и AI-модуль встроены в архитектуру не как вспомогательные элементы, а как часть ядра системы. Роль пользователя влияет на набор доступных операций. Журнал действий фиксирует ключевые события и обеспечивает аудит. Бизнес-процессы реализованы через отдельные API-действия. AI-модуль связан с записью клиента и сохраняет результат в отдельной сущности `Aidata`.

С точки зрения базы данных архитектура также остается предметно-ориентированной. Центральным объектом является запись `Appointment`, с которой связаны клиент, мастер, услуга, статус, транзакции и AI-данные. Отдельные сущности ролей, прав доступа и журнала действий позволяют поддерживать нормализованную структуру данных и делают систему расширяемой.

Воспроизводимый запуск backend обеспечивается за счет фиксированного набора зависимостей, миграций базы данных, seed-команды для демонстрационных данных и инструкции запуска. Это позволяет развернуть проект на другом компьютере и получить рабочее серверное приложение с тем же набором сущностей и API.

## 1.12. Тестирование

Для данной системы предусмотрены функциональное, интеграционное и ручное прикладное тестирование. Проверка считается успешной, если API корректно принимает и валидирует данные, права доступа реально ограничивают операции по ролям, бизнес-процессы проходят от начала до конца без нарушения логики, а AI-эндпоинт возвращает понятный результат и сохраняет его в базе данных.

Пройденными считаются такие проверки, при которых:

1. данные успешно создаются, изменяются и читаются через API;
2. ошибочные запросы возвращают ожидаемые сообщения об ошибках;
3. пользователь с неподходящей ролью не получает доступ к запрещенным операциям;
4. запись нельзя создать на занятое время;
5. AI-прогноз сохраняется и доступен для просмотра;
6. журнал действий пополняется после ключевых операций.

# 2. ПРОЕКТИРОВАНИЕ ИНФОРМАЦИОННОЙ СИСТЕМЫ

## 2.1. Архитектурное решение

Серверная часть реализована как отдельное Django-приложение. Она включает:

1. слой моделей;
2. слой сериализаторов;
3. слой представлений и API-маршрутов;
4. слой аутентификации и permission-классов;
5. слой служебной документации API.

Такое разделение делает проект понятным, масштабируемым и удобным для сопровождения.

## 2.2. Диаграмма вариантов использования

В системе можно выделить три основных актора: клиент, мастер и администратор. Клиент регистрируется, авторизуется, просматривает услуги и создает запись. Мастер просматривает расписание и свои записи. Администратор управляет сущностями, подтверждает операции, просматривает журнал действий и запускает анализ данных.

В отчет Word желательно вставить диаграмму вариантов использования с такими действиями:

1. регистрация;
2. авторизация;
3. просмотр профиля;
4. просмотр услуг;
5. создание записи;
6. изменение статуса записи;
7. проведение оплаты;
8. просмотр журнала;
9. запуск AI-анализа.

## 2.3. Диаграммы активности

Первая диаграмма активности должна описывать процесс создания записи: ввод данных, проверка времени, расчет окончания услуги, сохранение записи и формирование ответа.

Вторая диаграмма активности должна описывать работу AI-модуля: получение признаков, валидация, расчет вероятности, сохранение результата в `Aidata`, запись в `AuditLog` и возврат ответа.

## 2.4. Диаграмма классов

Диаграмма классов должна отражать основные сущности и связи между ними: `User`, `Role`, `AccessRight`, `Service`, `Status`, `MasterSchedule`, `Appointment`, `Transaction`, `Aidata`, `AuditLog`. Особое внимание следует показать связи роли с пользователем, записи с клиентом, мастером, услугой и статусом, а также зависимость AI-данных и транзакций от записи.

## 2.5. ER-модель базы данных

ER-модель строится вокруг сущности `Appointment`, которая связывает основные данные предметной области. Отдельные справочники ролей и статусов позволяют избежать дублирования и делают структуру нормализованной. Наличие журнала действий и AI-данных расширяет базовую модель и делает ее пригодной не только для учета, но и для аналитики.

## 2.6. Диаграмма компонентов

На диаграмме компонентов следует показать:

1. клиентскую часть или API-клиент;
2. Django backend;
3. базу данных SQLite.

Между клиентом и сервером используется HTTP, между сервером и БД — обращение через ORM. Для демонстрации backend используется также Swagger UI и Django admin.

# 3. РЕАЛИЗАЦИЯ ИНФОРМАЦИОННОЙ СИСТЕМЫ

## 3.1. Структура проекта

Проект организован по модульному принципу. В каталоге `config` находятся настройки проекта, маршрутизация и схема документации API. В каталоге `apps/users` размещены модели ролей, пользователей, прав доступа, а также сериализаторы и представления для аутентификации. В каталоге `apps/salon` реализованы модели и API для услуг, статусов, расписаний, записей, транзакций, AI-данных и журнала действий.

## 3.2. Реализация базы данных

База данных реализована средствами Django ORM. Используется SQLite, что упрощает локальный запуск и тестирование. На уровне моделей заданы ограничения, необходимые для предметной области: уникальность названий ролей и услуг, обязательность ряда полей, контроль временных интервалов и проверка пересечений записей.

Особенно важной частью является модель `Appointment`, которая автоматически рассчитывает время окончания записи на основе длительности услуги и запрещает наложение записей одного мастера на пересекающееся время.

## 3.3. Реализация серверной части

Серверная часть реализована как REST API. Для каждой основной сущности предусмотрены сериализаторы и viewset-ы. В системе доступны маршруты для регистрации, входа, обновления токена, получения профиля, а также для работы с ролями, пользователями, услугами, статусами, расписаниями мастеров, записями, транзакциями, AI-данными и журналом.

Отдельно реализованы бизнес-операции:

1. подтверждение записи;
2. отмена записи;
3. завершение записи;
4. оплата;
5. прогнозирование неявки клиента.

Во всех ключевых операциях формируются записи в `AuditLog`, что позволяет отслеживать действия пользователей и результаты операций.

## 3.4. Реализация аутентификации и прав доступа

Для авторизации используется JWT. Пользователь получает токен при входе и передает его в запросах к API. Разграничение прав построено на роли пользователя. Администратор имеет расширенные возможности управления, мастер работает со своим расписанием и своими записями, клиент ограничен собственными данными и операциями, относящимися к его записям.

## 3.5. Реализация AI-модуля

AI-функция реализована через эндпоинт `/api/appointments/{id}/predict-no-show/`. На вход передается набор признаков, связанных с поведением клиента и параметрами будущего визита. Сервер вычисляет итоговую вероятность неявки, сохраняет ее в `Aidata` и возвращает пользователю в процентах, например `3.90%`.

Практическая ценность функции состоит в том, что администратор может заранее оценить надежность записи и, например, принять решение о дополнительном подтверждении визита.

## 3.6. Документация API

Для проекта подготовлены OpenAPI-схема и Swagger-страница. Это позволяет просматривать список эндпоинтов, параметры запросов и ответы. Такой подход делает backend удобным для проверки, тестирования и дальнейшей интеграции с пользовательским интерфейсом.
